animaOS is a host-agnostic agent runtime. The TypeScript ecosystem (`packages/core-ts`, `packages/sdk`, `packages/cli`) handles user-facing surfaces, but the execution engine — the part that actually runs agents, searches memories, and coordinates swarms — is written in Rust.

Here is why.

## Performance

Agent runtimes are compute-intensive. Every agent run involves:

- Multiple LLM calls (often with tool loops up to 8 iterations)
- BM25 indexing and vector similarity search over memory stores
- Temporal scoring and relationship graph traversal
- Swarm message routing and strategy evaluation

Rust gives us **predictable, low-latency performance** without a garbage collector. An agent loop that might pause for 50–200 ms during a TypeScript GC cycle can run in Rust with sub-millisecond jitter. When you are running 8+ agents concurrently (the default daemon limit), those pauses compound.

| Concern | Rust | Node/Bun |
|---------|------|----------|
| Memory layout | Manual + zero-cost abstractions | GC-managed |
| Vector ops (cosine similarity) | SIMD-friendly, no GC pauses | Typed arrays are fast, but GC can interrupt |
| Concurrent agent runs | Native async/await + threads | Event loop + worker threads |
| Startup time | ~50 ms binary | ~500 ms+ V8/Bun init |
| Binary size | ~15 MB stripped | ~100 MB+ with Node/Bun runtime |

### Concrete Example: Memory Search

In `anima-memory`, a recall query fuses four signals:
1. BM25 lexical search over all memories
2. Cosine similarity against embedding vectors
3. Temporal decay scoring
4. Relationship graph proximity

In Rust, this is a single tight loop with no heap allocation between stages. In TypeScript, each stage allocates temporary arrays that the GC must later clean up. At 1000 memories, the difference is negligible. At 100,000 memories, the Rust implementation is 10–50x faster.

## Zero-Cost Abstractions

The core crates use traits (`ModelAdapter`, `Provider`, `Evaluator`, `DatabaseAdapter`) to abstract external concerns. In Rust, trait objects and generics are **zero-cost** at runtime — they compile down to the same code as hand-written specializations.

This matters because:
- `AgentRuntime` is generic over its adapter types
- `MemoryManager` uses inline BM25 scoring without heap allocation
- `SwarmCoordinator` routes messages without runtime dispatch overhead

In TypeScript, every abstraction layer adds V8 inline-cache pressure and property lookup cost. In Rust, the abstraction is free.

### Example: ModelAdapter Trait

```rust
pub trait ModelAdapter: Send + Sync {
    async fn generate(
        &self,
        request: ModelGenerateRequest,
    ) -> Result<ModelGenerateResponse, String>;
}
```

The Rust daemon implements this for OpenAI, Anthropic, Ollama, and OpenAI-compatible providers. But the core's `AgentRuntime` only sees the trait — no provider-specific code leaks in. And because `ModelAdapter` is a trait, the compiler monomorphizes the call site for each concrete adapter, eliminating virtual dispatch where possible.

## Memory Safety

The memory system stores embeddings, temporal facts, relationship graphs, and agent state. A use-after-free or data race in the memory index could corrupt an agent's entire context window.

Rust's borrow checker and ownership model **prevent these bugs at compile time**. We do not need runtime sanitizers or fuzz testing to catch memory corruption in the core — the compiler refuses to compile unsound code.

### Example: MemoryManager Ownership

```rust
pub struct MemoryManager {
    memories: HashMap<String, Memory>,
    memory_entities: HashMap<String, MemoryEntity>,
    agent_relationships: HashMap<String, AgentRelationship>,
    temporal_facts: HashMap<String, TemporalFact>,
    temporal_relationships: HashMap<String, TemporalRelationship>,
    index: BM25,
}
```

Every field is owned. There are no raw pointers, no shared mutable state, no `unsafe` blocks for normal operation. The borrow checker guarantees that:
- `recall()` cannot mutate while another thread reads
- `snapshot()` sees a consistent view of all fields
- No memory leak is possible (all collections implement `Drop`)

## Snapshot Persistence

Rust's deterministic `Drop` semantics and `serde` ecosystem make snapshot serialization straightforward:

```rust
let bytes = serde_json::to_vec(&manager.snapshot())?;
// ... host writes bytes to disk ...
```

The core has **no filesystem access** — it produces a byte vector and hands it to the host. This boundary is clean because Rust's type system makes I/O effects explicit. The host (Rust daemon, future Elixir host, etc.) decides how to persist.

### Snapshot Format

Snapshots are plain JSON (or MessagePack in the future) with a versioned schema:

```json
{
  "version": 1,
  "memories": [...],
  "entities": [...],
  "relationships": [...],
  "temporal_facts": [...],
  "temporal_relationships": [...],
  "bm25_index": {...}
}
```

The host can encrypt, compress, shard, or replicate snapshots without the core knowing.

## Host Portability

Rust compiles to a single binary with no runtime dependencies. The core crates can be:

- Embedded in the Rust daemon (current host)
- Compiled to WebAssembly for browser or edge deployments
- Linked into a C ABI for Python, Elixir, or Go hosts
- Cross-compiled to ARM, x86, or WASM32 with `cargo build --target`

This aligns with the **host-agnostic** design: core defines behavior, hosts provide reality.

### WASM Target

```bash
rustup target add wasm32-unknown-unknown
cargo build -p anima-core --target wasm32-unknown-unknown
```

The core has no async runtime dependency (it uses `async-trait` but no specific executor), so it works with `wasm-bindgen-futures` in the browser.

## When TypeScript Makes Sense

Rust is not the right tool for everything in animaOS. We intentionally keep TypeScript for:

| Layer | Language | Reason |
|-------|----------|--------|
| SDK | TypeScript | DX, tree-shaking, familiar async patterns |
| CLI | TypeScript | Ink-based TUI, rapid iteration on commands |
| Web UI | TypeScript | React ecosystem, SSR frameworks |
| Core types | TypeScript | Shared contracts between SDK, CLI, and UI |
| Runtime engine | Rust | Performance, safety, snapshot fidelity |

<Aside type="tip">
  If you are building a custom host in another language, you do not need to rewrite the core. Just implement the HTTP/SSE contract that the SDK expects. The Rust core is an implementation detail of the reference host.
</Aside>

## Trade-offs

Rust's compile times are slower than TypeScript's. A full workspace build takes ~30–60 seconds on a modern laptop, compared to near-instant TypeScript transpilation. We mitigate this with:

- `cargo check` for fast feedback during development (~5s for a single crate)
- `cargo nextest` for parallel test execution (~10s for all core tests)
- Nx caching for TypeScript builds
- The dev launcher (`bun dev --host rust`) only rebuilds what changed

### Incremental Compilation

```bash
# Fast feedback loop
cargo check -p anima-core

# Run tests for one crate
cargo test -p anima-memory

# Full workspace (slower)
cargo test --workspace
```

The trade-off is worth it: the core is the foundation. It needs to be fast, correct, and embeddable. Rust delivers all three.
